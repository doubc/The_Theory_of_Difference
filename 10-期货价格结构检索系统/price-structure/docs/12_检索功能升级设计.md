# 检索功能升级设计 v1.0

> 研究工作台 UI 检索能力补全：当日机会扫描 + 历史精细检索
> 基于 v3.0 工作台现状审计，2026-04-23

---

## 一、问题陈述

### 1.1 现状审计

当前工作台（`app.py` v3.0）的 5 个 Tab 中，检索相关功能存在三个断裂：

| Tab | 功能 | 状态 | 问题 |
|-----|------|------|------|
| 📡 今天值得关注什么 | 当日结构展示 | ⚠️ 部分 | 只展示当前选定品种，无跨品种扫描；无机会排序；无关注度评分 |
| 🔍 历史对照 | 历史相似检索 | ⚠️ 部分 | 检索条件粗糙（仅颗粒度/范围/年数）；无法按方向/反差/运动态/价位过滤 |
| 📊 跨品种对比 | 品种间对比 | ✅ 完整 | — |
| 🗺️ 稳态地图 | 稳态视图 | ✅ 完整 | — |
| 📝 复盘日志 | 日志记录 | ✅ 完整 | — |

### 1.2 核心缺失

**缺失 A：当日机会检索**
- 后端已有：`scripts/daily_scan.py`（全市场扫描）、`scripts/daily_scan_v3.py`、`src/retrieval/opportunity.py`（机会聚合 + 关注度评分）
- 前端未接入：用户无法在 UI 中触发全市场扫描并查看排序后的机会清单

**缺失 B：历史检索条件不足**
- 当前只有 3 个控件：检索颗粒度（粗/中/细）、检索范围（单品种/全品种）、年数滑块
- 缺少：日期范围、价位范围、方向、反差类型、运动状态、最小相似度、排序方式

**缺失 C：检索结果过滤**
- 检索结果是一次性展示，无法在结果中二次过滤
- 缺少：方向筛选、相似度排序、中位数收益统计

---

## 二、设计原则

1. **不改底层引擎** — `src/retrieval/`、`src/compiler/`、`src/models.py` 不动
2. **复用已有能力** — `opportunity.py` 的 `compute_attention_score`、`aggregate_opportunity` 直接调用
3. **渐进增强** — 保留现有 UI 结构，在现有 Tab 内增加控件
4. **文档先行** — 本文档定义清楚后再改代码

## 三、新增定义

### 3.1 当日机会扫描（Daily Opportunity Scan）

**触发方式**：Tab 1 顶部增加 "🔍 全市场扫描" 按钮

**扫描流程**：
```
用户点击扫描
  → 遍历 ALL_SYMBOLS（MySQL + CSV + symbol_meta）
  → 对每个品种：
      load_bars(symbol) → compile_full(bars, config)
      → 对每个编译出的结构：
          - 过滤：只保留 t_end 在最近 N 天内的结构
          - 计算 motion、projection、contrast
          - 如果有历史样本库：retrieve() → aggregate_opportunity()
          - 如果无样本库：用 invariants 直接计算关注度
  → 按 attention_score 降序排列
  → 展示 Top 10
```

**输出对象**：`DailyOpportunity`（复用 `opportunity.py` 的 `Opportunity` 结构）

**关注度评分公式**（已有，来自 `opportunity.py`）：
```
attention = sim_total × 40 + direction_conf × 25 + min(potential/0.20, 1) × 25 + min(count/5, 1) × 10
```

**缓存策略**：`@st.cache_data(ttl=300)` 缓存扫描结果，避免重复编译

### 3.2 历史检索条件（Search Filters）

**新增控件清单**：

| 控件 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `date_range` | `st.date_input` | (最近3年, 今天) | 历史案例的时间范围 |
| `zone_price_range` | `st.slider` | (自动推断) | Zone price_center 的价位范围 |
| `direction_filter` | `st.multiselect` | ["up", "down", "unclear"] | 只看特定方向的案例 |
| `contrast_filter` | `st.multiselect` | 全选 | 按反差类型过滤 |
| `motion_filter` | `st.multiselect` | 全选 | 按运动状态过滤 |
| `min_similarity` | `st.slider` | 0.3 | 最低相似度阈值 |
| `sort_by` | `st.radio` | "相似度" | 结果排序方式 |
| `outcome_window` | `st.slider` | 30 | 后续表现计算窗口（天） |

**过滤逻辑**（在检索完成后、展示前应用）：
```
results = [r for r in all_candidates
           if r.similarity >= min_similarity
           and r.direction in direction_filter
           and r.contrast_type in contrast_filter
           and r.motion_tendency in motion_filter
           and date_start <= r.t_end <= date_end
           and zone_min <= r.zone_center <= zone_max
          ]
```

### 3.3 检索结果二次过滤

在 Tab 2 检索结果展示区增加实时过滤栏：
- 方向筛选：`st.multiselect`（up/down/unclear）
- 排序切换：`st.radio`（相似度/后续涨幅/日期）
- 统计面板增强：增加中位数收益、上涨/下跌概率

## 四、数据流

### 4.1 当日机会扫描数据流

```
ALL_SYMBOLS (发现品种)
  ↓
[symbol₁, symbol₂, ...]
  ↓  load_bars() + compile_full()
Structure[] per symbol
  ↓  过滤最近 N 天
recent_structures[]
  ↓  similarity() + retrieve() [可选]
TemplateMatch[] or direct invariants
  ↓  aggregate_opportunity() / compute_attention_score()
Opportunity[] (含 attention_score)
  ↓  按 attention_score 排序
Top 10 DailyOpportunity
  ↓  渲染为卡片
UI
```

### 4.2 历史检索条件数据流

```
用户设置筛选条件
  ↓
search_params = {
  date_range, zone_price_range, direction_filter,
  contrast_filter, motion_filter, min_similarity, sort_by
}
  ↓
现有检索逻辑 (Tab 2 的 run_search)
  ↓
all_candidates[] (未过滤)
  ↓  应用 search_params 过滤
filtered_candidates[]
  ↓  按 sort_by 排序
sorted_results[]
  ↓  二次过滤 (方向/相似度)
display_results[]
  ↓  渲染
UI (含增强统计面板)
```

## 八、风险与约束

1. **全市场扫描性能**：品种数可能 50+，每个品种编译耗时 ~0.5s，总计 ~25s。用进度条 + 缓存缓解。
2. **MySQL 依赖**：扫描依赖 MySQL 连接。MySQL 不可用时只扫描 CSV 品种，需明确提示。
3. **内存**：同时编译多品种的 Structure 对象可能占用较多内存。用逐品种流式处理，不全部驻留。
4. **缓存失效**：`@st.cache_data(ttl=300)` 5 分钟过期。用户点击"刷新"按钮时清除全部缓存。
