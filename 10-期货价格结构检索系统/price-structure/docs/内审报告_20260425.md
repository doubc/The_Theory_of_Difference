## 专家一：逻辑架构师 — 逻辑一致性审查

### 🔴 问题 1.1：compute_motion 的投票语义仍然模糊

`compute_motion()` 用 4 个信号投票决定 `phase_tendency`。虽然已将 `→breakdown` 改为 `→breakout`，但**信号 D（守恒通量）的逻辑仍有矛盾**：

```python
# 信号 D: 守恒通量方向（权重 0.20）
if flux > 0.3:
    # 差异释放 → 可能走向 breakout
    votes["→breakout"] = ...  # ← flux 正 = 差异释放 = 突破？
elif flux < -0.3:
    # 差异压缩 → 走向 confirmation
    votes["→confirmation"] = ...  # ← flux 负 = 差异压缩 = 确认？
```

**矛盾**：在信号层（`signals.py`），`flux > 0` 且方向一致 = 好事（突破确认）。但在 `compute_motion` 里，`flux > 0` 投给 `→breakout`（突破），`flux < -0.3` 投给 `→confirmation`（确认）。这意味着：
- 价格缓慢上行 + flux 负（差异压缩）→ `→confirmation`（确认）→ ✅ 趋势确认
- 价格快速突破 + flux 正（差异释放）→ `→breakout`（突破）→ 📈 趋势上行

**实际上这是对的**，但容易让人误解"突破"是坏事。建议：在 `qualitative_judgment` 的注释中明确 `→breakout = 价格正在突破zone`，不是"崩塌"。

**建议**：在 `compute_motion` 的 docstring 中补充语义表。

### 🔴 问题 1.2：qualitative_judgment 的 "趋势破坏" 与 "假突破反转" 重叠

```python
# 当前逻辑：
if "breakout" in tendency:
    if flux > 0:
        return "趋势上行"
    else:
        return "趋势破坏"  # ← 这个

# 信号层：
if signal.kind == "fake_breakout":
    return "假突破反转"  # ← 和这个有什么区别？
```

**矛盾**：`→breakout` + 负 flux = "趋势破坏"，但 `FAKE_BREAKOUT` 信号也是突破后反转。两者可能同时触发，输出哪个取决于优先级。

**建议**：`qualitative_judgment` 中，当信号是 `FAKE_BREAKOUT` 时，应覆盖运动态判断。当前代码已这样做了（信号优先级最高），但 "趋势破坏" 这个阶段名应改为 "突破失败" 以区分。

### 🟡 问题 1.3：quality.py 的 breakdown dict 命名冲突

`quality.py` 中有 `breakdown = {}`（质量评分分解），与运动态的 `breakout` 容易混淆。

**建议**：重命名为 `score_breakdown` 或 `dimension_scores`。

## 专家三：性能工程师 — 性能审查

### 🔴 问题 3.1：DTW 相似度 O(n×m) 暴力计算

`similarity.py` 中的 DTW 实现是标准 O(n×m)，当候选池有 1200+ 结构时，每次检索都要对所有候选做 DTW。

**当前复杂度**：检索一次 = 1200 × O(cycle_count²) ≈ 1200 × 25 = 30,000 次内循环

**建议**：
1. 对 speed_ratio 序列做 SAX（Symbolic Aggregate approXimation）降维，先粗筛再精算
2. 或用 `src/fast/_dtw.c` 的 C 扩展（已有，需确认是否启用）

### 🟡 问题 3.2：tab_scan.py 每次 rerun 都重新编译

Streamlit 的 rerun 机制下，`_scan_all_symbols()` 每次交互都会重新编译所有品种。

**建议**：用 `@st.cache_data(ttl=300)` 缓存编译结果，只在点击"刷新"时清除。

### 🟡 问题 3.3：compute_motion 中的列表推导式

```python
recent_srs = [c.speed_ratio for c in s.cycles[-3:]]
exit_distances = [abs(e.end.x - zone.price_center) / max(zone.bandwidth, 1e-9) for e in recent_exits]
```

对每个结构都要遍历 cycles 列表。当批量处理 65 个品种 × 多个结构时，可以接受，但可以优化为在编译时预计算。

**建议**：将 `phase_tendency` 的计算移到 `compile_full()` 中，避免重复计算。

## 专家四：UX 设计师 — 用户体验审查

### 🔴 问题 4.1：卡片信息过载

`tab_scan.py` 单张卡片包含：品种名、Zone 价位、Cycle 数、运动态、通量、质量分、风险等级、时效标签、信号、定性判断、止损提示。用户需要在密集的 HTML 中寻找关键信息。

**建议**：
1. 主卡片只显示 3 个核心指标：**定性判断 + 现价 + 信号**
2. 详细信息折叠在 expander 中
3. 用颜色编码替代文字标签（红=危险，绿=安全，黄=观望）

### 🟡 问题 4.2：定性判断的置信度对用户意义不大

`📈 趋势上行 (差异释放中，上行趋势延续·80%)` — 用户不关心"差异释放中"这种技术术语。

**建议**：改为用户语言：
- `📈 上行趋势中 · 动能充足 · 可跟随`
- `🔻 顶部反转信号 · 建议观望`
- `🔄 区间震荡 · 等待突破方向`

### 🟡 问题 4.3：缺少全局概览

用户打开工作台，需要逐个 Tab 查看。缺少一个"一页概览"展示所有品种的当前状态。

**建议**：在 Tab 1 顶部增加一个全局状态摘要：
```
今日市场概览：
  📈 上行趋势: CU, AL (2个)
  🔄 震荡整理: RB, I, MA (3个)  
  🔻 反转信号: AU (1个)
  🔵 形成中: 12个品种
```

## 专家五：交易策略师 — 交易辅助审查

### 🔴 问题 5.1：止损位只是文字提示，没有数值计算

```python
stop_loss_hint=f"Zone.{'upper' if price_position == 'above' else 'lower'} {'+' if price_position == 'below' else '-'} 0.3×bandwidth"
```

这输出的是 `Zone.upper - 0.3×bandwidth` 这样的公式字符串，不是实际价格。用户需要自己算。

**建议**：直接输出数值：
```python
stop_loss = upper - 0.3 * bandwidth  # 直接算出来
stop_loss_hint=f"止损 {stop_loss:.1f} (Zone上沿下方0.3倍带宽)"
```

### 🔴 问题 5.2：缺少盈亏比计算

系统输出了止损位和仓位系数，但没有计算**盈亏比（Risk/Reward Ratio）**。交易者最关心的是：这笔交易值不值得做？

**建议**：在 Signal 对象中增加 `rr_ratio` 字段：
```python
# 目标位 = Zone 对面边界 + 突破幅度
target = zone.opposite_boundary + breakout_depth
rr_ratio = abs(target - entry) / abs(entry - stop_loss)
```

### 🟡 问题 5.3：缺少交易计划摘要

当前信号输出是分散的（信号类型 + 方向 + 止损 + 仓位系数），用户需要自己拼装成交易计划。

**建议**：增加 `generate_trade_plan()` 函数，输出一句完整的交易指令：
```
CU2507 做多 | 入场 78500 | 止损 77800 (Zone下沿) | 目标 80200 (前高) | 仓位 60% (B层) | 盈亏比 2.4:1
```

### 🟡 问题 5.4：假突破后的反向入场缺少具体点位

`FAKE_BREAKOUT` 信号说"假突破·反向做多/做空"，但没说在哪里入场、止损放哪。

**建议**：在 `detect_fake_breakout()` 返回值中增加 `entry_price` 和 `stop_loss_price`。
