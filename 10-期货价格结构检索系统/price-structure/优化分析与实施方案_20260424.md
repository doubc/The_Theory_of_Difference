## 二、发现的问题（按严重度排序）

### 🔴 P0: 代码结构问题

#### 问题 1: models.py 前向引用未正确处理

`Structure` 类使用了 `MotionState`、`ProjectionAwareness`、`StabilityVerdict`，但这些类在 `Structure` 之后才定义。Python dataclass 在运行时需要实际类型来做 `to_dict()`/`from_dict()`，虽然 `from __future__ import annotations` 延迟了类型解析，但以下场景会出问题：

```python
# models.py line ~230
@dataclass
class Structure:
    motion: MotionState | None = None  # MotionState 在 50 行后才定义
    projection: ProjectionAwareness | None = None
    stability_verdict: StabilityVerdict | None = None
```

**影响**：`from_dict()` 反序列化时如果 dict 中有 `motion` 字段，需要 `MotionState` 类已经存在。当前代码能工作是因为 Python 的 late binding，但这是脆弱的。

**修复**：将所有辅助类（MotionState, ProjectionAwareness, StabilityVerdict, SystemState）移到 Structure 之前定义，或使用字符串类型注解。

#### 问题 2: models.py 和 relations.py 职责混乱

`models.py` 中定义了 `first_diff`, `log_diff`, `second_diff`, `time_gap`, `distance_to_zone`, `relative_distance_to_zone`, `extrema_dispersion`, `extrema_similarity` 等算子函数。同时 `relations.py` 也定义了 `segment_speed_ratio`, `segment_time_ratio` 等算子。

**问题**：
- `extrema_similarity()` 是一个 DTW 相似度函数，放在 `models.py` 中不合适
- `models.py` 的注释说"对象只保存状态，不做复杂计算"，但实际包含了大量计算
- `relations.py` 的注释说"复用 models.py 中定义的基础算子"，但边界不清

**修复**：
- `models.py` 只保留 dataclass 定义 + 最简单的属性计算（如 `Point.log_x`）
- 所有算子函数统一移到 `relations.py`
- `extrema_similarity()` 移到 `retrieval/similarity.py`

#### 问题 3: check_conservation() 中的 channel_scores 语义不一致

```python
# zones.py 中 _infer_contrast 的返回值语义是"反差类型"
# 但 check_conservation() 中 channel_scores 的 key 是通道名（"zone_bandwidth", "speed_ratio"）
# 而 transfer_channels 列表中存放的是补偿性变化的标识（"speed_ratio_expanding"）
# 这与 _infer_contrast 的 ContrastType 枚举完全不是同一套语义
```

**影响**：`detect_stability_illusion()` 从 `conservation["transfer_channels"]` 取值加入 `pending_channels`，但这些值（如 `"speed_ratio_expanding"`）与 ProjectionAwareness 的 `blind_channels`（如 `"shorter_timeframe"`, `"volume"`）语义不同，混在一起造成混淆。

**修复**：统一通道语义——`transfer_channels` 只存放目标维度（`"shorter_timeframe"`, `"volume"`, `"oi"`），补偿性变化作为 `channel_scores` 的正值信号。

### 🟡 P1: 理论-代码偏差

#### 偏差 A: 反差类型推断过于简陋

`_infer_contrast()` 只看时间跨度和试探密度，把 180 天以上的都归为 OVERSUPPLY，30 天内 3 次以上的都归为 PANIC。

**理论要求**（V1.6 定义 2.2）：共同反差是"多个差异单元共同面对的**外部差异**"。这意味着需要考虑：
- 宏观事件（金融危机、政策变化）
- 品种特性（农产品的季节性 vs 金属的周期性）
- 市场状态（牛市末端 vs 熊市末端）

**改进建议**：
1. 引入品种元数据中的行业分类
2. 结合 Zone 位置相对于历史价格分位数
3. 结合成交量变化模式
4. 允许用户手动标注覆盖

#### 偏差 B: 守恒检查的"影子层"表述与实现不一致

`check_conservation()` 的 docstring 说"不声称差异转到了某社会差异维度，只观测价格特征A被压缩时价格特征B是否在补偿性变化"。但代码中 `flux_score` 的计算只是简单地取各通道信号的平均值，没有真正验证"A 被压缩时 B 在扩张"这个守恒关系。

**改进建议**：
1. 对每对通道 (A, B)，计算 A 的变化与 B 的变化的相关系数
2. 如果 A 下降而 B 上升（负相关），才是真正的守恒转移
3. 当前的实现更像是"异常检测"而非"守恒验证"

#### 偏差 C: 运动态的阶段转换过于依赖速度比

`compute_motion()` 中的阶段判断完全基于最后 3 个 cycle 的 speed_ratio 趋势：
- sr_trend > 0.5 → breakdown
- sr_trend < -0.3 → confirmation
- 方向反转 → inversion

**理论要求**（V1.6 Ch5）：阶段转换应该考虑：
- Zone 的完整性是否在变化
- 试探密集度是否在加速
- 守恒通量的方向
- 稳态距离的变化

**改进建议**：多信号投票机制，而非单一指标阈值。

### 🟢 P2: 工程质量改进

#### 改进 1: 重复代码

- `_safe_cv()` 在 `relations.py` 和 `quality.py` 中重复
- `extrema_dispersion()` 的逻辑在 `models.py` 和 `relations.py` 的 `_safe_cv()` 中重复
- DTW 的 Python fallback 在 `similarity.py` 和 `models.py` 的 `extrema_similarity()` 中重复

#### 改进 2: 类型注解不一致

部分使用 `list[X]`（Python 3.9+），部分使用 `List[X]`。部分函数参数用 `list`（无类型），如 `check_conservation(bars: list | None)`。

#### 改进 3: 测试覆盖不足

`tests/` 目录有 7 个测试文件，但：
- 没有测试 P0 新增的 `_infer_contrast()`, `_find_nearest_stable()`, `check_conservation()`
- 没有测试 P1 新增的 `compute_liquidity_stress()`, `compute_fear_index()`, `compute_time_compression()`
- 没有测试 `quality.py` 的分层逻辑
- 没有测试 `lifecycle.py` 的追踪逻辑

#### 改进 4: 配置管理

`CompilerConfig` 的参数（如 `min_amplitude=0.02`, `cluster_eps=0.015`）是硬编码默认值，没有解释为什么选这些值。应该：
- 在 docstring 中说明参数的典型范围和选择依据
- 提供不同品种的推荐配置（金属 vs 农产品 vs 能化）
